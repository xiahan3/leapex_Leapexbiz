# 老客首页 · 待办事项 展示规则与场景枚举

> 适用：Leapex 利柏思 微信小程序 · 老客版首页（「我的公司」页）「待办事项」模块
> 目标：把「用户当前需要做什么、系统在等什么」一屏说清；用**优先级 + 配色 + 一句话文案 + 直达跳转**驱动用户完成关键动作，降低流失。
> 关联页面：订单列表 / 订单详情、账单支付、KYC 认证、公司注册（NNC1）、各服务办理流程、消息中心、我的公司。

---

## 一、设计原则

1. **可行动**：每条待办 = 一个明确的下一步动作 + 一个直达入口（点击即到对应页面/步骤）。
2. **分级不刷屏**：按紧急度分三级配色；同一公司/同一订单的待办就近聚合，避免堆叠。
3. **状态驱动、自动消解**：待办由订单/KYC/公司生命周期的**状态**实时计算生成；对应动作完成后**自动消失**，无需用户手动关闭。
4. **不制造焦虑**：营销类/周期提醒类默认弱化；只有「阻断类」用红色。
5. **合规优先**：KYC / AMLO / 签字类待办不可被跳过或永久忽略。

---

## 二、优先级与视觉规范（三级 + 信息级）

| 级别 | 含义 | 配色（原型 token） | 图标示例 | 是否可忽略 | 排序权重 |
|---|---|---|---|---|---|
| **P0 阻断** | 不处理流程无法推进 / 合规风险 | 红 `--red` / `--red-bg` | `ti-alert-circle` `ti-credit-card-off` `ti-user-x` | 否 | 100 |
| **P1 待处理** | 需用户行动、尚未阻断但应尽快 | 橙黄 `--orange` / `--yellow-bg` | `ti-file-upload` `ti-receipt` `ti-signature` | 否（可延后） | 60 |
| **P2 提醒** | 周期性 / 到期类，非紧急 | 蓝 `--blue` / `--blue-bg` | `ti-calendar-due` `ti-refresh` | 可延后/免打扰 | 30 |
| **P3 信息** | 完成通知 / 营销 / 优惠 | 灰绿 `--text3` / `--green-bg` | `ti-gift` `ti-download` | 可关闭 | 10 |

**每条待办的标准结构（数据模型）**

```
Todo {
  id            // 唯一标识
  level         // P0 | P1 | P2 | P3
  type          // 见「七、场景枚举」的 type 枚举
  companyId     // 归属公司（可为空 = 账户级）
  companyName   // 展示用
  orderNo       // 关联订单（可空）
  title         // 主文案（动作导向，≤ 16 字）
  desc          // 副文案（对象 + 原因/进度，≤ 30 字）
  icon          // Tabler 图标名
  colorToken    // 配色档（跟随 level）
  target        // 跳转：{page, params}
  dueAt         // 截止/到期时间（P2/P3 用于排序与升级）
  createdAt
  dismissible   // 是否可延后/关闭（P0=false）
  resolveOn     // 自动消解条件（状态达成即移除）
}
```

## 三、待办生成规则（按生命周期）

待办**不是**手工维护的列表，而是对以下几条状态机的实时投影。任一状态推进都会**重算**待办集合。

### 3.1 订单 / 支付生命周期
`待支付 → 待提交信息 → 待 KYC → 待人工预审 → 办理中 → 已完成 / 已退回 / 已取消`

| 订单当前状态 | 生成待办 | 级别 |
|---|---|---|
| 待支付（新下单） | 账单待支付 | P1 |
| 支付发起后失败 / 未完成 | 支付失败 · 请重新支付 | **P0** |
| 已支付、未提交办理信息 | 提交办理信息 | P1 |
| 已提交信息、未做 KYC | 完成 KYC 认证 | P1 |
| 需签字文件未回寄 | 确认并寄回签字文件 | P1 |
| 人工预审被退回 | 注册资料被退回 · 需修改 | **P0** |
| 办理中（政府审批） | —（无待办，进度在订单页/公司卡展示） | — |
| 已完成 | 交付物已就绪 · 可下载（一次性） | P3 |
| 已退回 / 已取消 | —（转入订单页历史） | — |

### 3.2 KYC 尽调生命周期
`未开始 → 进行中（分步：AML说明/身份/地址/业务声明/资金来源）→ 待人工审核 → 通过 / 驳回 / EDD`

| KYC 状态 | 生成待办 | 级别 |
|---|---|---|
| 已支付但未开始 KYC | 完成 KYC 认证 | P1 |
| 进行中，某步未完成（如地址证明未传） | 补充 {缺失项}（如「补充地址证明」） | P1 |
| 人工审核驳回 | KYC 未通过 · 需修改资料 | **P0** |
| 命中 PEP / 制裁名单，需补充说明（EDD） | 补充合规说明（EDD） | **P0** |
| 通过 | —（消解相关待办） | — |

### 3.3 公司注册（NNC1）流程
`核名 → 支付 → 提交注册信息（公司资料/成员董事/组织章程/人工预审）→ 政府审批 → 交付 CI/BR`

| 情形 | 生成待办 | 级别 |
|---|---|---|
| 核名结果不可用（与已注册近似/含敏感词） | 核名未通过 · 需改名重核 | **P0** |
| 核名通过但未继续下单注册 | 核名已通过 · 去下单注册 | P1 |
| 注册信息填写中断（有草稿/暂存） | 继续填写注册信息 · 第 N 步 | P1 |
| 人工预审退回（信息不符 NNC1） | 注册资料被退回 · 需修改 | **P0** |

### 3.4 存续期 · 周期性提醒（年审 / 记账报税 / 商标 / BR）
- **年审（NAR1，成立周年日 + 42 天内）：递进式多轮提醒**
  - T-90 天：第 1 轮 · P2 蓝
  - T-60 天：第 2 轮 · P2 蓝（加粗）
  - T-30 天：第 3 轮 · P1 橙
  - T-7 天：第 4 轮 · **P0 红**（临期）
  - 逾期：P0 红 + 标注「已逾期 · 政府罚款递增」
- **记账报税**：按账期，报税季前 X 天生成「本年度审计/报税待办」P2。
- **利得税报税表 PTR**：IRD 发出后倒计时提醒 P2 → 临期 P1/P0。
- **商标续期**：到期前 6 个月 P2 → 3 个月 P1。
- **商业登记证 BR 续期**：到期前提醒 P2。

### 3.5 账户 / 合规
| 情形 | 待办 | 级别 |
|---|---|---|
| 我的公司档案不完整（缺 CI/BR/成立日期） | 补充公司资料 | P1 |
| 优惠券即将过期（≤ 7 天且可用于在办服务） | 优惠券即将过期 · 去使用 | P3 |
| 证件到期（KYC 留存的身份证/护照过期） | 更新身份证件 | P1 |

## 四、排序、聚合、去重、折叠

1. **排序**：先按 `level`（P0 > P1 > P2 > P3）降序；同级按 `dueAt` 升序（越临期越靠前）；再按 `createdAt`。
2. **同公司聚合**：同一公司的多条待办在视觉上归到该公司名下（副文案带公司名）；若一家公司待办 ≥ 3 条，折叠为「XX 公司 · N 项待办 ›」，点击展开。
3. **去重 / 合并**：
   - 同一订单同时「待支付」不会既出「账单待支付」又出「支付失败」——按最新状态取其一。
   - 同一 KYC 的多个缺失步骤合并为一条「补充 KYC 资料（缺 X 项）」，点击进 KYC 未完成的第一步。
4. **上限与「查看全部」**：首页最多展示 **5–6 条**（P0/P1 优先占位）；超出折叠为底部「查看全部待办（N）›」跳待办中心/消息中心。
5. **P0 置顶常驻**：任意 P0 未消解时，模块标题旁显示红点计数；P0 不参与折叠隐藏。

---

## 五、空状态与全部完成

- **无任何待办**：模块显示轻量空态——`✓ 暂无待办，一切正常` + 一句引导（如「可去『服务』看看年审 / 记账」）。不显示红点。
- **仅剩 P2/P3**：正常展示，模块标题不显示红点（红点只统计 P0/P1）。
- **首次进入 / 新客转老客**：若刚完成首单支付，优先展示「完成 KYC 认证」引导，弱化其他。

## 七、状态流转示意（单订单视角）

```
下单
 └─ 未支付 ───────────────▶ [8 账单待支付 P1] ──支付失败──▶ [1 支付失败 P0]
      支付成功
 └─ 待提交信息 ───────────▶ [9 提交办理信息 P1]
      信息已提交
 └─ 待 KYC ───────────────▶ [10 完成 KYC P1] ──缺步──▶ [11 补充资料 P1] ──驳回──▶ [2 KYC 未通过 P0]
      KYC 通过
 └─ 待签字（如需） ───────▶ [7 确认并寄回签字文件 P1]
      文件递交
 └─ 人工预审 ─────────────▶ 预审退回 ──▶ [4 注册资料被退回 P0]
      预审通过 → 政府审批（无待办，进度展示）
 └─ 已完成 ───────────────▶ [20 交付物已就绪 P3]（一次性）
```
> 每一步的待办在进入下一步时**自动消解**；任何环节出现驳回/失败即升级为对应 P0。

## 八、多公司维度

- 待办**按公司归集**；一个账户下多家公司时，副文案统一带公司名前缀，便于区分。
- 「我的公司」卡片状态与待办联动：如「星辰科技 · 注册进行中 · 第 3/6 步」对应正有 `info_pending / kyc_pending` 待办。
- 账户级待办（无 companyId，如优惠券过期、口令码优惠）排在公司类待办之后。

## 九、与其他模块的关系

| 模块 | 关系 |
|---|---|
| **消息中心** | 待办的「提醒类（P2/P3）」同步下发一条消息；年审递进提醒每轮各发一条。待办是「当前要做」，消息是「历史通知流」。 |
| **订单页 / 订单详情** | P0/P1 多来自订单状态；点击待办直达订单对应步骤，进度只读联动后台服务管理。 |
| **后台（管理端）** | 待办由后台的订单状态、KYC 审核结论、人工预审/办理退回、到期规则驱动；后台一改状态，小程序待办即重算。 |

## 十、边界与异常

1. **同时多条 P0**：全部展示，不折叠；按 `dueAt` 与业务阻断度排序（支付/KYC/核名优先于提醒类退回）。
2. **待办与实际状态不一致（弱网/缓存）**：以服务端状态为准，进入页面时刷新重算；避免「点进去发现已办完」。
3. **已取消 / 已退款订单**：不再产生待办，相关待办立即消解。
4. **免打扰**：P2/P3 支持「稍后提醒 / 本轮不再提示」，但年审临期（T-7）与 P0 不可免打扰。
5. **到期口径**：年审以「公司成立周年日」为基准 + 42 天法定期限；PTR 以 IRD 发出日为基准；均以官方为准，前端仅做倒计时提醒。
